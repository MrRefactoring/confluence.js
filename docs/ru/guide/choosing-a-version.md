# Выбор версии

У Confluence Cloud два REST API, и `confluence.js` поставляет оба. Эта страница — короткий ответ на вопрос, какой вам нужен.

## Короткий ответ

**Берите v2.** Его развивает Atlassian, у него чище ответы, и все новые эндпоинты появляются в нём.

**Берите v1 только для того, чего в v2 нет** — список ниже, и это вся причина, по которой v1 ещё здесь.

## Что покрывает v2

Страницы, блог-посты, custom content, доски (whiteboards), базы, папки, smart links, комментарии (footer и inline), вложения (чтение), метки (чтение), задачи, версии, пространства, свойства пространств, свойства контента, права пространств (чтение), роли пространств, пользователи, уровни классификации, политики данных.

```ts
import { createV2Client } from 'confluence.js';

const confluence = createV2Client({ host, auth });

await confluence.page.getPages({ spaceId: ['123456'] });
```

## Что есть только в v1

| Область | Почему не v2 |
|---|---|
| **Загрузка вложений** | v2 читает вложения, но не умеет их создавать |
| **Ограничения доступа к контенту** | эквивалента в v2 нет вообще |
| **Подписки (watches)** | наблюдатели страниц, пространств и меток — только v1 |
| **Поиск CQL** | в v2 нет языка запросов |
| **Проверка прав** | `checkContentPermission` — «может ли этот пользователь сделать X» |
| **Состояния контента** | «В работе», «Готово к ревью», … |
| **Группы** | список групп и их участников |
| **Связи (relations)** | избранное и любые связи между сущностями |
| **Шаблоны и blueprints** | шаблоны контента — только v1 |
| **Темы, оформление, настройки пространства** | внешний вид сайта и пространств |
| **Свойства пользователя** | произвольный JSON на пользователе |
| **Аудит** | журнал аудита сайта |
| **Долгие задачи** | очередь за асинхронными операциями |
| **Перемещение, копирование, архивация страниц** | v2 читает дерево, но не перестраивает его |
| **Конвертация тела** | рендер storage → export_view |

```ts
import { createV1Client } from 'confluence.js';

const confluence = createV1Client({ host, auth });

await confluence.search.searchByCQL({ cql: 'type=page and label="release-notes"' });
```

## Один host, обе версии

Обе фабрики принимают **один и тот же голый URL сайта**. Путь к API принадлежит запросу, а не конфигу, поэтому при переключении версии в `host` ничего не меняется:

```ts
const host = 'https://your-domain.atlassian.net';

const v2 = createV2Client({ host, auth });
const v1 = createV1Client({ host, auth });

const page = await v2.page.getPageById({ id: 12345 });
const restrictions = await v1.contentRestrictions.getRestrictions({ id: '12345' });
```

Использовать обе рядом — нормально и ожидаемо: читайте через v2, а за тем, чего в нём нет, идите в v1.

::: tip Идентификаторы — те же самые
Id страницы в v2 и id контента в v1 — одно и то же число. v2 типизирует его как `number`, v1 — как `string`. Вся конвертация сводится к `String(page.id)`.
:::

## v1 — дополнение, и он сжимается

v1 — это не законсервированный старый API. `confluence.js` генерирует его из **опубликованной** спеки v1, а Atlassian продолжает её урезать по мере роста v2: 37 операций, которые были в `confluence.js@2` — среди них `getContent`, `createContent`, `getSpace` — из неё уже исчезли, потому что их покрывает v2.

Это направление, а не случайность. Ожидайте дальнейшего сужения v1 и переносите каждый вызов в v2, как только там появится эквивалент. [Руководство по миграции](../migration/v2-to-v3) сопоставляет каждую удалённую операцию с заменой.

## Один клиент или два?

Два клиента ничего не стоят — это обычные объекты над общим транспортом. Но если хочется один, импортируйте плоские функции и передавайте им общий core-клиент:

```ts
import { createClient } from 'confluence.js/core';
import { getPages } from 'confluence.js/v2';
import { searchByCQL } from 'confluence.js/v1';

const client = createClient({ host, auth });

const pages = await getPages(client, { spaceId: ['123456'] });
const hits = await searchByCQL(client, { cql: 'type=page' });
```

Это же та форма, которая tree-shaking'уется — см. [Tree-Shaking](./tree-shaking).
